02 AI-DLC: Planner Document Artifacts

바이브 코딩에서 가장 중요한 것은 프롬프트(설계 문서) 작성입니다.

이 프로젝트에서 전체 2일 중 8시간 이상을 PRD부터 API Design까지의 설계 문서 작성에 투자했습니다. AI에게 "이거 만들어줘"라고 한 줄 던지는 것이 아니라, PRD → Userflow → Data Model → Architecture → API Design → ADR의 AI 개발 라이프사이클(AI-DLC)을 따라 16개 문서(~430KB)와 22개 ADR을 작성했습니다. 이 문서들이 곧 Generator에게 전달되는 프롬프트이며, 코딩 중 설계 판단이 불필요하도록 모든 결정을 사전에 완료했습니다.

Why Prompt Engineering is Everything

바이브 코딩의 성패는 AI에게 무엇을 어떻게 만들어야 하는지를 얼마나 정밀하게 전달하느냐에 달려 있습니다. runs/planner/ 폴더에 있는 16개 문서는 단순한 기획서가 아니라, AI가 읽고 실행할 수 있도록 설계된 구조화된 프롬프트입니다. 테이블 DDL, TypeScript 인터페이스, MCP 도구 매핑, SSE 시퀀스 다이어그램까지 — Generator가 "어떻게 할까?"라고 고민할 여지를 제로로 만드는 것이 목표였습니다.

결과: 7개 스프린트 전원 2~3회 시도 내 PASS, 평균 34분/Sprint. 설계 문서에 투자한 시간이 구현의 속도와 품질을 결정했습니다.

03 Three-Terminal Architecture

이 프로젝트의 또 다른 핵심은 3개의 독립된 Claude Code 터미널을 동시에 운영한 것입니다.
Anthropic Harness 논문(Prithvi Rajasekaran)의 GAN 영감 — 생성자가 자기 코드를 평가하면 관대해지는 편향을 제거하기 위해, 프롬프트를 작성하는 에이전트, 코드를 작성하는 에이전트, 평가하는 에이전트를 물리적으로 다른 터미널에서 분리했습니다.

Terminal 1 — Planner (Prompt)
# 역할: 설계 문서 작성 + Sprint Contract 생성
# 도구: Claude Code (claude)

$ claude
> PRD 작성, Userflow 설계, Data Model 정의
> Architecture 설계, API Design
> 6라운드 크로스 검증 (100+ 불일치 수정)
> Sprint Contract 7개 + Prompt 7개 생성

# 산출물: runs/planner/ (16개 문서, ~430KB)
# 산출물: runs/sprint-contracts/ (7+7개)
Terminal 2 — Generator (Code)
# 역할: Sprint Contract 읽고 코드 구현
# 도구: Claude Code (claude)

$ claude
> sprint-contract.md를 읽는다
> Backend (Express) + Frontend (React) 구현
> TypeScript 빌드 확인
> git commit

# Contract만 읽음 — 설계 결정 없음
# Sprint당 평균 34분
Terminal 3 — Evaluator (Score)
# 역할: Sprint Contract 기준으로 독립 채점
# 도구: agentinc evaluator agent

$ claude
> Generator 코드를 모름 — Contract만 기준
> Major/Minor 이슈 발견 → evaluations/ 기록
> PASS → handoff 생성
> FAIL → Generator에게 수정 요청

Terminal 1 sprint-contracts/*.md  →  Terminal 2 src/**/*.ts(x)  →  Terminal 3 evaluations/*.md   |   FAIL → 수정 후 재채점  |  PASS → handoffs/*.md → 다음 Sprint

Why Three Terminals?

자기 평가 편향 제거: Generator가 스스로 "잘 했다"고 판단하는 것을 원천 차단. Evaluator는 Generator의 코드 작성 과정을 전혀 모르고, Contract 기준으로만 채점합니다.

컨텍스트 분리: 각 터미널은 독립된 컨텍스트 윈도우를 사용. Planner의 설계 맥락이 Generator의 구현을 오염시키지 않고, Evaluator는 편견 없이 순수하게 결과물만 봅니다.

파일 기반 소통: 에이전트 간 직접 대화 없음. 오직 sprint-contracts/, evaluations/, handoffs/ 파일을 통해서만 소통합니다.

06 Lessons Learned

Lesson 1
PRD is a Journey, Not a Document
조원들과 PRD를 논의하며 Pain Point를 수집하는 과정은 AM으로서의 일상을 되돌아보는 즐거운 시간이었습니다. "이게 불편했지", "이건 왜 항상 수작업이지?" — 매일 겪던 문제를 언어화하는 것 자체가 솔루션의 시작이었습니다.
Lesson 2
Plan Exhaustively, Execute Mechanically
8시간의 설계가 4시간의 거의 완벽한 구현을 가능하게 했습니다. Generator는 설계 판단 없이 Contract만 실행했습니다.
Lesson 3
Separate Generator and Evaluator Physically
모든 Sprint에서 최소 1건의 Major 이슈가 발견되었습니다. 같은 에이전트가 평가했다면 놓쳤을 것입니다. 3개 터미널 분리가 핵심이었습니다.
Lesson 4
Cross-Document Validation is Non-Negotiable
12개 문서에서 100건 이상의 불일치. 검증 없이 바로 코딩했다면 디버깅에 훨씬 더 많은 시간이 소요되었을 것입니다.
